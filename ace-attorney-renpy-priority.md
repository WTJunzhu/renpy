# Ace Attorney Ren'Py — 特化开发优先级

> 目标：在 Ren'Py 引擎基础上，构建一个可制作逆转裁判系列同人游戏的特化工具。
> 覆盖范围：逆转裁判1~6 + 逆转检事1~2 + 大逆转裁判 的核心机制。

---

## 模块总览

```
P0  文本演出层          [中]
P0  法庭/询问系统（融合逆转检事） [大]
P0  搜证系统（逆转检事风格）     [大]
P1  心灵枷锁系统         [中]
P1  角色动画状态机        [中]
P1  案件数据架构 + 模板     [中]
P2  逻辑象棋（Mind Chess）    [大]
P2  高潮推理（Revisualization） [中大]
P2  高级演出效果         [小中]
P3  可视化编辑器         [很大]
P3  Launcher 特化 + 素材包   [中]
```

## P0 — 必须实现（核心体验，没有这些就不是逆转裁判）

### P0-1 文本演出层 `[中]`

逆转裁判的"味道"首先来自文本表现，这是玩家接触的第一个差异化点。

| 子项 | 说明 | 复杂度 |
|------|------|--------|
| 逐字打字机音效 | 每显示一个字符触发一次短促嘟嘟声，音色随角色变化。需要 hook Ren'Py 的文本渲染管线，在 `Text` Displayable 的逐字输出阶段插入音频触发 | 中 |
| 同一句内变速 | 在文本中标记速度区间，如 `{speed=30}慢慢说{/speed}{speed=80}突然加速{/speed}`。需自定义 TextTag | 小 |
| "异议/待った/くらえ" 全屏冲击 | 全屏角色立绘 + 放大震动 + 画面闪白 + 音效，约1秒后消失。需自定义 Displayable，可参考 Ren'Py 的 `transition` 机制 | 中 |
| 关键台词停顿 | 自定义 TextTag `{aa_pause=0.5}`，在指定位置自动暂停而非等待点击 | 小 |
| 情绪文字效果 | 文字震动/放大/变色，通过 TextShader 或逐字符 Displayable 实现 | 中 |
| 嘟嘟声角色化 | 不同角色（辩护方、检察方、证人、法官）使用不同音色的嘟嘟声 | 小 |

### P0-2 法庭/询问系统（融合逆转检事）`[大]`

这是核心玩法循环。需要融合逆转检事系列的特性：询问环节不仅出现在法庭，也可以在搜证过程中随时插入。

| 子项 | 说明 | 复杂度 |
|------|------|--------|
| 证言数据结构 | 每条证言 = 多条陈述，每条陈述含：文本、角色立绘状态、威慑后追加文本/新陈述、正确举证物（支持单证物或双证物组合）、举证成功后跳转、举证失败扣血量 | 中 |
| 证言展示模式 | 逐条展示 + 节奏停顿 + "叮"音效 + 顶部进度指示点 + 左右切换 | 中 |
| 询问交互 UI | 在证言展示上叠加威慑/举证按钮，切换到证物选择面板。需完整的 Screen 设计 | 大 |
| 威慑机制 | 每条陈述可威慑 → 触发对话 → 可能追加新陈述（证言列表动态变化）→ 可能扣血 → 可能获得新证物 | 中 |
| 举证机制 | 选择证物 → 比对当前陈述 → 成功：异议动画 + 推进剧情 / 失败：扣血动画 | 中 |
| 血槽系统 | 顶部血槽（通常5格），扣血动画（红色闪烁 + 减少 + 音效），归零 Game Over。需自定义 Displayable | 中 |
| 双证物组合举证 | 某些矛盾需同时出示两件证物。举证面板需支持"选择第一件 → 选择第二件"的流程 | 中 |
| 证言回溯 | 随时查看已解锁的完整证言列表 | 小 |
| **逆转检事特性：任意阶段插入询问** | 搜证过程中与NPC对话时可触发询问环节（不只是法庭阶段）。询问的数据结构和UI复用，但触发时机和上下文不同 | 中 |
| **逆转检事特性：证言分支** | 威慑某条证言可能导致证言走向不同分支，而非简单追加。需要证言树而非证言列表 | 中 |

### P0-3 搜证系统（逆转检事风格）`[大]`

以逆转检事系列为蓝本：更强调场景内自由探索、逻辑推理、与角色的深度交互。

| 子项 | 说明 | 复杂度 |
|------|------|--------|
| 可点击热点 | 场景中特定区域可点击调查，鼠标悬停有高亮/提示框。Ren'Py 有 `imagemap` 但需要扩展以支持动态高亮和条件触发 | 中 |
| 角色移动 | 玩家控制角色在场景中移动（WASD/方向键），靠近热点时出现交互提示。需要自定义实时移动 Displayable，参考 `renpy/display/dragdrop.py` | 大 |
| 地点导航 | 地图界面选择目的地，某些地点在获得特定证物/触发特定事件后才解锁 | 中 |
| NPC 交互 | 与NPC对话时的多级菜单：闲聊（多个话题）/ 出示证物 / 询问（触发法庭式询问环节）。不同证物触发不同反应和解锁 | 中 |
| 证物发现 | 调查热点后自动获得新证物，加入证物列表，有发现动画和音效 | 小 |
| 证物检查 | 证物详情界面，支持多角度图片查看（3D旋转为可选增强） | 小 |
| 搜证→询问无缝切换 | 逆转检事的核心体验：搜证过程中随时可以进入询问环节，询问结果反过来影响搜证进度 | 中 |
| 场景条件解锁 | 某些热点/NPC/地点需要满足特定条件（拥有某证物、完成某对话、询问成功）才会出现 | 中 |

## P2 — 增强（提升品质和多样性）

### P2-1 逻辑象棋（Mind Chess）`[大]`

逆转检事2中御剑怜侍的特殊能力。限时心理博弈，模拟象棋般的对弈思维。

| 子项 | 说明 | 复杂度 |
|------|------|--------|
| 对弈 UI | 棋盘式界面，御剑与对手两侧，中间对话区，底部选项区 | 大 |
| 倒计时系统 | 实时倒计时，增加紧迫感。Ren'Py 不原生支持实时计时，需用 `renpy.timeout` 或自定义事件循环 | 中 |
| 攻防选项 | 每轮提供多个"攻击"和"防御"选项，正确选择推进博弈，错误选择失去"宝石"（类似血量） | 中 |
| 博弈阶段 | 多轮对弈，每轮对手的防御模式不同，需要观察和推理 | 中 |
| 宝石/生命系统 | 类似血槽但视觉风格不同，失去宝石有独特动画 | 小 |
| 胜负判定 | 成功：解锁新证词/线索。失败：扣血后重试或 Game Over | 小 |
| 数据结构 | 定义对弈场景：对手、轮次、每轮的选项+正确答案+错误反应+推进对话 | 中 |

## 已确定的设计决策

> 以下为开发前已敲定的架构决策，实现时遵循此节。

### 决策 1：集成方式 — Library 为主，Fork 为辅

- 优先通过 `renpy/common/` 放置扩展模块（自定义 Statement、Screen、Displayable、Python 类）
- 仅在 Ren'Py 确实不支持 hook 的地方（如文本逐字渲染回调）做最小化核心修改
- 所有 AA 专用代码放在独立的文件中（如 `00aa_courtroom.rpy`、`00aa_investigation.rpy`），不混入 Ren'Py 原有模块

### 决策 2：脚本语言 — .rpy 为主，JSON 为辅

- **对话/演出/流程控制**：使用 Ren'Py 原生 .rpy 脚本，通过自定义 Statement 扩展语法
- **证物/角色/案件元数据**：使用 JSON 定义（方便可视化编辑器生成）
- **Excel 写剧情**：Ren'Py 不原生支持 Excel 导入（仅有导出为 .tab 的配音工作流），如有需要可在 P3 阶段开发 Excel→.rpy 转换器
- 自定义 Statement 设计原则：**细粒度为主**，让剧情语言与动作可以逐行交错推进、最大化沉浸感；仅在大的结构性环节（如 `cross_examination` 包裹整个询问过程）使用粗粒度

**自定义 Statement 候选清单（细粒度示例）**：
```renpy
# 粗粒度：包裹一个完整环节
cross_examination "证人A的证言":
    ...

# 细粒度：每个操作独立
begin_testimony "证人A"                    # 开始证言展示
stmt "那天晚上我在现场。"                    # 一条证言陈述
stmt "我看到了被告。"                        # 下一条
press 1                                     # 威慑第1条
    "你确定看清了吗？"                       # 威慑后的追问对话
    add_stmt "呃...其实我不太确定。"          # 追加新证言
present 1 using 碎玻璃证物                   # 对第1条举证
    jump caught_lying                        # 成功跳转
penalty 1                                   # 扣1格血
end_testimony                               # 结束证言

# 搜证中的细粒度操作
hotspot "花瓶" pos (800, 400) radius 60      # 定义可调查热点
examine "花瓶"                               # 调查热点
    "花瓶碎了一地，碎片散落在地板上。"
    get_evidence 花瓶碎片                     # 获得证物
talk 糸锯圭介                                # 与NPC对话
    topic "关于案发时间"                      # 闲聊话题
    present 花瓶碎片                          # 出示证物
```

### 决策 3：画面分辨率 — 1920×1080 (16:9)

- 统一使用 1920×1080 作为设计基准
- 不支持多分辨率模板（不做 GBA 像素风适配）
- 所有 UI、背景、角色立绘按此分辨率设计

### 决策 4：搜证角色移动 — 混合模式 + 手柄预留

- **鼠标**：点击场景中某处，角色自动走过去；点击热点直接交互
- **键盘**：WASD/方向键实时控制角色移动
- **手柄**：预留接口，左摇杆移动 + 按键交互，具体适配延后
- 实现方式：自定义 Displayable，处理键盘事件 + 碰撞检测 + 鼠标点击

### 决策 5：数据管理 — 独立 CourtRecord 类

- 使用 `renpy.revertable.RevertableObject` 子类集中管理证物/角色/状态
- 内部使用 `RevertableDict` / `RevertableList`，自动参与 Ren'Py 的存档和回滚
- 无需自定义序列化代码，Ren'Py 的 pickle 机制自动处理
- 数据分层：
  - **Tier 1（RevertableObject）**：证物列表、角色档案、询问状态、血量 → 参与回滚和存档
  - **Tier 2（NoRollback）**：UI 状态（滚动位置、面板开关） → 存档但不回滚
  - **Tier 3（persistent）**：已解锁的成就/画廊/完成度 → 跨存档保留
  - **Tier 4（_constant=True）**：临时动画状态、缓存 → 运行时专用

### 决策 6：第一阶段示例案件 — 混合 AA1-1 + AAI1-1

为最大化测试覆盖，示例案件从两个来源各取一部分：

**来自逆转裁判1第一章（仅法庭部分）**：
- 证言展示 + 逐条切换 + 进度指示点
- 威慑机制（追问证人）
- 举证机制（出示证物指出矛盾）
- 血槽系统（扣血 + Game Over）
- "异议！"全屏冲击动画
- 法官互动（敲锤、宣布休庭）

**来自逆转检事1第一章（仅搜证部分）**：
- 场景内角色移动（WASD + 鼠标点击）
- 可点击热点调查 + 证物发现
- NPC 对话（闲聊话题 + 出示证物）
- 地点导航（地图切换）
- 搜证过程中插入询问环节

**不包含**：心灵枷锁、逻辑象棋、高潮推理（这些是 P2 阶段的内容）

### 决策 7：兼容性 — 保留 Ren'Py 全部功能

- 不精简 Ren'Py 原有功能（NVL模式、普通视觉小说模式、所有内置 Screen）
- AA 专用模块作为扩展叠加在 Ren'Py 之上
- 创作者可以在同一个项目中混合使用普通 Ren'Py 和 AA 功能

### 素材统计汇总

| 类别 | 预估数量 | 获取难度 |
|------|----------|----------|
| 嘟嘟声（5种角色音色） | 5~10个 | 低（合成即可） |
| 语音（异议/待った/くらえ） | 4~6个 | 中（需录制或合成） |
| 音效（UI/系统/演出） | 30~50个 | 低（免费音效库可覆盖大部分） |
| BGM | 8~15首 | 中（需寻找免费/CC协议音乐） |
| 法庭/UI 背景图 | 10~15张 | 高（需原创绘制） |
| 搜证地点背景 | 每案件5~15张 | 高（需原创绘制或照片改编） |
| UI 元素（按钮/面板/血槽等） | 30~50张 | 中（可设计一套后批量复用） |
| 角色立绘（每角色） | 8~20帧 | 高（核心素材，决定视觉品质） |
| 特殊效果（心锁/象棋/推理） | 15~30张 + 动画帧 | 中（可简化后迭代） |
| **总计（最小可用版本）** | 约100~150个素材文件 | — |

> **建议**：第一阶段使用简化的占位素材（纯色方块、简单几何图形、合成音效），优先让系统跑通。视觉品质在第二阶段集中提升。角色立绘可先使用社区共享的素材或简笔画风格。

## 开发顺序建议

```
第一阶段：核心循环（能做出一个可玩的简单案件）
  P0-1 文本演出层
  P0-2 法庭/询问系统（先实现基础版，不含逆转检事扩展）
  P0-3 搜证系统（先实现基础版，热点+地点导航+NPC对话）
  P1-3 案件数据架构（最小可用版本）

第二阶段：演出品质（让体验像逆转裁判）
  P1-2 角色动画状态机
  P2-3 高级演出效果
  P0-1 文本演出层增强（情绪文字、变速等）

第三阶段：特殊系统（丰富玩法多样性）
  P1-1 心灵枷锁系统
  P2-2 高潮推理
  P2-1 逻辑象棋

第四阶段：逆转检事融合（扩展玩法深度）
  P0-2 增强：逆转检事特性（任意阶段插入询问、证言分支）
  P0-3 增强：搜证→询问无缝切换、场景条件解锁

第五阶段：工具化（让非程序员也能用）
  P1-3 增强：完整模板 + 多章节管理
  P3-2 Launcher 特化 + 素材包
  P3-1 可视化编辑器
```
