## 第一阶段：核心循环（能跑通一个简单案件）

### Step 1：设计文档 — 自定义语法 + JSON Schema + 文件结构 `✅`

**产出文件**：[ace-attorney-design.md](ace-attorney-design.md)

产出三份设计文档，供后续所有步骤参照：

1. **自定义 Statement 语法规范**
   - 精确定义每个语句的名称、参数、是否接受 block、执行时行为
   - 覆盖：`begin_testimony` / `stmt` / `press` / `present` / `penalty` / `end_testimony` / `begin_cross_examination` / `hotspot` / `examine` / `talk` / `topic` / `get_evidence` / `move`
   - 给出完整示例脚本片段

2. **JSON 数据格式 Schema**
   - `evidence.json`：证物定义（id、名称、描述、图标、检查图、组合关系）
   - `characters.json`：角色定义（id、名称、立绘路径、嘟嘟声音色、动画集）
   - `locations.json`：地点定义（id、名称、背景图、热点列表、NPC列表）
   - `case.json`：案件元信息（章节结构、阶段跳转）

3. **文件结构**
   - 确定 AA 模块在 `renpy/common/` 中的文件命名和组织方式
   - 确定游戏项目模板的目录结构（`game/` 下的文件组织）

预计：1 步

---

### Step 2：CourtRecord 数据类 `✅`

**产出文件**：[renpy/common/00aa_core.rpy](renpy/common/00aa_core.rpy)

实现证物/角色/状态的集中管理，是后续所有系统的数据基础。

- 定义 `Evidence` 类（RevertableObject 子类）
- 定义 `Profile` 类
- 定义 `CourtRecord` 类（RevertableObject 子类，内部用 RevertableDict/RevertableList）
- 提供接口：`add_evidence` / `remove_evidence` / `update_evidence` / `has_evidence` / `add_profile` / `get_evidence` / `get_profile`
- 提供 flag 管理（通用状态标记，用于条件解锁等）
- 验证：存档/读档/回滚正常工作

预计：1 步

---

### Step 3：自定义 Statement 注册框架 `✅`

**产出文件**：[renpy/common/00aa_statements.rpy](renpy/common/00aa_statements.rpy)

搭建 Ren'Py 自定义语句的注册基础设施。

- 封装 `renpy.register_statement` 的调用模式
- 实现 Statement 执行时的上下文管理（跳转、block 执行、参数解析）
- 先实现 1~2 个简单语句验证框架可用（如 `get_evidence`、`penalty`）
- 确保自定义语句与 Ren'Py 原有语法（jump、call、if、menu 等）无缝混用

预计：1 步

---

### Step 4：文本演出基础 `✅`

**产出文件**：[renpy/common/00aa_text.rpy](renpy/common/00aa_text.rpy)

**备注**：嘟嘟声采用循环播放模式（begin→slow_done/end），播放期间持续发声。逐字触发需要引擎级修改，留作后续增强。

实现逆转裁判标志性的文本表现。

- 逐字打字机音效：hook Ren'Py 文本渲染管线，每显示一个字符触发一次音频
- 嘟嘟声角色化：不同角色使用不同音色（通过 Character 参数指定）
- 自定义 TextTag：`{aa_pause=N}` 自动停顿 N 秒
- 自定义 TextTag：`{aa_speed=N}` 变速
- 验证：与 Ren'Py 原有的 `{w}`、`{p}`、`{fast}` 等标签兼容

预计：1~2 步

---

### Step 5：法庭 UI 基础 `✅`

**产出文件**：[renpy/common/00aa_screens.rpy](renpy/common/00aa_screens.rpy)

用 Ren'Py Screen Language 构建法庭相关的 UI 面板。

- **血槽 Screen**：顶部显示，5格，支持扣血动画（红色闪烁 + 减少），归零触发 Game Over
- **证言面板 Screen**：底部证言文本区 + 进度指示点（当前高亮）+ 左右切换
- **证物选择面板 Screen**：半透明遮罩 + 证物网格 + 选中高亮 + 确认/取消
- **询问按钮栏 Screen**：威慑/举证按钮，仅在询问模式下显示
- 所有 Screen 支持显示/隐藏/切换动画

预计：1~2 步

---

### Step 6：证言展示系统 `✅`

实现证言从展示到交互的完整流程。

- `begin_testimony` 语句：初始化证言上下文
- `stmt` 语句：逐条展示证言，每条之间有节奏停顿 + "叮"音效
- `end_testimony` 语句：结束证言，跳转到下一阶段
- 证言数据结构：支持 `testify` block 内定义多条 stmt
- 证言面板联动：进度指示点随当前 stmt 同步
- 左右键切换证言条目

预计：1~2 步

---

### Step 7：询问交互系统 `✅`

实现威慑/举证/扣血的核心玩法循环。

- `press` 语句：威慑当前证言 → 执行 block 内对话 → 可追加 stmt
- `present` 语句：打开证物选择 → 举证 → 比对正确性 → 成功跳转 / 失败扣血
- `penalty` 语句：扣 N 格血 + 扣血动画 + 音效
- 血量归零 → Game Over 画面
- `begin_cross_examination` 语句：包裹整个询问流程，自动管理 UI 状态切换
- 证言分支：`press` 导致 stmt 列表变化（追加/替换/跳转）

预计：2~3 步

---

### Step 8：搜证基础 `✅`

实现搜证场景的热点交互。

- `hotspot` 语句：定义可调查区域（位置、大小、触发条件）
- `examine` 语句：调查热点 → 显示对话 → 可触发 `get_evidence`
- 热点高亮：鼠标悬停时的视觉提示（光圈/边框）
- 角色移动（鼠标）：点击场景中某处，角色自动走过去（transform + ease）
- 角色移动（键盘）：WASD/方向键实时控制，自定义 Displayable 处理键盘事件和碰撞检测
- 角色移动（手柄）：预留接口，左摇杆映射到移动方向

预计：2~3 步

### 问题分支

- **问题**：`examine` 块内的 `say` 语句通过 `renpy.execute()` 执行，可能无法正确显示对话文本并等待玩家点击。`renpy.execute()` 的行为在自定义语句上下文中未经充分验证。
- **排查**：需要运行时测试。如果 `renpy.execute(say_node)` 不显示对话，需改用 `renpy.call()` + 动态标签或其他机制。
- **解决方案**：待测试。备选方案：将 `examine` 改为非块语句，对话通过 Python 函数 `renpy.say()` 实现。
- **影响**：如果需要修改，仅影响 `next_investigate` 中的 examine 块执行逻辑，不影响 hotspot/SceneDisplayable 设计。

---

### Step 9：搜证 — NPC 交互 + 地点导航 `✅`

实现搜证中与角色对话和地点切换。

- `talk` 语句：与 NPC 对话的入口
- `topic` 语句：闲聊话题列表，已聊过的话题标记
- `present`（搜证版）：向 NPC 出示证物，不同证物触发不同反应
- `move` 语句：切换到另一个地点
- 地图 Screen：显示可用地点列表，已解锁/未解锁状态
- 地点条件解锁：拥有某证物/完成某对话后解锁新地点

预计：2~3 步

### 问题分支

- **问题**：与 Step 8 相同，`topic` 块和 `present`（搜证版）块内的 `say` 语句通过 `renpy.execute()` 执行，可能无法正确显示对话。
- **影响**：talk 语句的默认反应使用 `renpy.say(None, ...)` 作为备选，可正常工作。topic/present 块的对话需运行时验证。

### Step 10：示例案件 — 端到端测试 `✅`

用 AA1-1 法庭 + AAI1-1 搜证的一小部分验证所有系统。

**AA1-1 法庭部分（选取一个完整证言+询问段）**：
- 证人登场 → 证言展示（4~5条 stmt）→ 询问（威慑 + 举证）→ 扣血 → 成功推进
- 包含至少一次 press 追加新证言
- 包含至少一次 present 成功和一次失败

**AAI1-1 搜证部分（选取一个场景）**：
- 一个地点，2~3 个可调查热点
- 1 个 NPC（3 个话题 + 2 种证物出示反应）
- 证物发现 → 证物检查
- 搜证完成后进入法庭

**验证项**：
- 存档/读档正常
- 回滚正常
- 所有 UI 正确显示和隐藏
- 音效/动画触发正确
- 流程完整可通关

预计：2~3 步

**测试文件**：
- JSON 数据：`game/aa/evidence.json`（5件证物）、`characters.json`（5个角色）、`locations.json`（3个地点）
- 测试脚本：`game/test_case.rpy`
- 项目配置：`game/options.rpy`、`game/gui.rpy`、`game/screens.rpy`

**覆盖功能清单**：
- 法庭部分：`begin_testimony`、`stmt`、`press`、`present`、`penalty`、`end_testimony`、`get_evidence`、血槽 UI、证言面板、证物选择
- 搜证部分：`investigate`、`hotspot`、`examine`、`end_investigate`、`get_evidence`、SceneDisplayable（移动+交互）
- NPC 对话：`talk`、`topic`、`present`（搜证版）、`set_flag`
- 地点切换：`move`

**已知待验证项**（与 Step 8/9 相同）：
- `examine`/`topic` 块内的 `say` 语句通过 `renpy.execute()` 执行的行为

## 第二阶段：演出品质（让体验像逆转裁判）

### Step 11：角色动画状态机 `⬜`

建立标准化的角色动画管理系统。

- 动画状态定义：idle / talk / objection / breakdown / sweat / slam / think / shock / smile / angry / sad
- 统一接口：`show character_name animation_name` 触发动画切换
- 内部管理帧序列、持续时间、循环/单次播放
- LayeredImage 或自定义 Displayable 实现（待设计文档确定方案）
- 每个状态支持 2~8 帧动画（角色立绘序列帧或部件切换）

预计：2~3 步

---

### Step 12：异议演出 — 全屏冲击动画 `⬜`

实现 "异议！""待った！""くらえ！" 的标志性演出。

- 全屏冲击 Displayable：角色立绘 + 冲击背景 + 文字缩放震动
- 画面闪白效果
- 配套音效触发
- 持续约 1 秒后自动消失，返回之前的画面状态
- 支持辩护方/检察方不同版本（不同背景色、不同立绘）
- 在脚本中通过自定义语句或函数调用触发

预计：1~2 步

### Step 13：高级演出效果集 `⬜`

- 聚光灯效果（证人席聚焦）

## 第三阶段：特殊系统（丰富玩法多样性）

### Step 14：心灵枷锁系统 `⬜`

- `psyche_lock` 语句：包裹整个心锁环节
- 多层锁环定义：每层绑定正确证物 + 错误反应
- 心锁 UI Screen：暗色遮罩 + 锁链视觉 + 锁环指示
- 答对：锁链断裂动画 + 音效 + 解锁下一层
- 答错：扣血 + 反馈对话
- 全部解开：揭示真相的特殊对话 + 成就感演出
- 心锁状态绑定到角色 flag（同一角色可有多段心锁对话）

预计：2~3 步

---

### Step 15：高潮推理系统 `⬜`

- `revisualization` 语句：包裹整个推理环节
- 推理 UI Screen：逻辑问题展示 + 证物/证词选择
- 逻辑链构建：多环节串联，每环选对进入下一环，选错扣血
- 视觉效果：线索飞入、逻辑链连接动画、真相揭示光效
- 最终揭示演出：全部推理正确后的特殊场景
- 配套 BGM 切换

预计：2~3 步

---

### Step 16：逻辑象棋系统 `⬜`

- `mind_chess` 语句：包裹整个对弈环节
- 棋盘 UI Screen：双方头像 + 对话区 + 选项区
- 倒计时系统：实时计时（renpy.timeout 或自定义事件循环）
- 攻防选项：每轮提供多个选项，正确推进/错误失去宝石
- 宝石/生命系统：碎裂动画 + 音效
- 多轮对弈：每轮对手防御模式不同
- 胜负判定：成功解锁线索 / 失败扣血重试

预计：3~4 步

## 第四阶段：逆转检事融合（扩展玩法深度）

### Step 17：搜证中插入询问 `⬜`

让询问系统不局限于法庭阶段。

- 在 `talk` 语句中支持 `cross_examination` 子块
- NPC 对话过程中可自然过渡到询问环节
- 询问 UI 在搜证场景中正确显示（不切换法庭背景）
- 询问结果（获得新证物/解锁新话题/推进剧情）反馈到搜证状态

预计：1~2 步

---

### Step 18：证言分支系统 `⬜`

让威慑操作可以改变证言走向。

- 证言数据结构从列表改为树（每条 stmt 的 press 可指向不同后续 stmt）
- UI 正确渲染分支后的证言序列
- 分支状态追踪（哪些分支已探索）
- 与 `present` 的交互：不同分支下正确举证可能不同

预计：2~3 步

---

### Step 19：搜证→法庭无缝过渡 `⬜`

管理阶段切换的状态保持和演出。

- 搜证阶段收集的证物/flag 正确传递到法庭阶段
- 过渡演出：转场动画 + 场景切换
- 法庭阶段获得的新证物可带回搜证阶段（如果设计允许）
- 多章节间的证物继承和状态持久化

预计：1~2 步
